# 工作流程规范

当用户要求你编写方案或代码时，必须严格遵循以下四步流程，不得跳过或合并步骤。

## 触发与不触发
- **触发**：编写方案、设计文档，或实现新功能 / 较大改动 / 重构。
- **不触发**：简单 bug 修复、代码解释、一般问答。

## 流程
1. **需求确认** → 2. **方案设计** → 3. **方案评审** → 4. **代码落地**

每一步须等用户明确回复"确认"或"通过"后，才可进入下一步。

```
- [ ] 1. 需求确认（需用户回复"确认"）
- [ ] 2. 方案设计
- [ ] 3. 方案评审（需用户回复"通过"）
- [ ] 4. 代码落地
```

---

## 1. 需求确认（需用户交互）
- 将你对需求的理解整理成结构化清单，包括：
  - 功能目标
  - 输入 / 输出
  - 边界与约束
  - 非功能需求（性能、安全、可维护性等）
  - 已有系统的影响范围
- 以列表或问答形式向用户确认，**必须等待用户明确回复"确认"或提出修改后，才可进入下一步**。
- 确认结果必须整理为文档，存放于 `docs/` 目录下，文件名建议为 `docs/需求确认-{简短描述}.md`。

## 2. 方案设计（AI 主导，关键节点需用户介入）
- 基于已确认的需求，给出至少 1 个可行方案。若存在明显不同 trade-off 的备选方案，应列出 2~3 个并对比。
- 方案必须包含以下维度（按自行判断的权重综合考量）：
  - **业务实现完整性**：能否满足所有需求与异常场景
  - **代码性能**：时间复杂度、资源消耗、并发处理等
  - **落地代价**：开发工作量、侵入性、依赖变更、测试成本、维护成本
- 如果各维度权重无法自行判断，必须**暂停并请用户介入裁定权重**。
- 所有涉及代码的方案，必须附上：
  - **类图**（Mermaid 或文字描述，标明关键类、职责、关系）
  - **流程图**（Mermaid 或文字描述，核心业务流程或数据流）
- 方案设计内容同样要输出为文档，存放于 `docs/方案设计-{简短描述}.md`。

## 3. 方案评审（需用户交互）
- 将方案设计文档的核心内容（含类图、流程图、权重与取舍说明）以简洁摘要呈现给用户。
- 明确指出需要用户决策的点，例如权重取舍、备选方案选择、风险接受度等。
- 等待用户明确回复"通过"或提出修改意见，修改后再次呈现，直至用户确认通过。

## 4. 代码落地
- 用户评审通过后，方可开始编写代码。
- 编码必须对齐最终确认的方案与设计图（类图、流程图）。
- 若编码过程中发现方案存在缺陷或新的重要取舍，必须回到第 2 步更新方案，并经第 3 步评审后再继续。
- 代码完成后，在 `docs/` 下补充必要的实现说明或接口文档（如需要）。

---

## 通用规则
- 所有文档使用 Markdown 格式，放置于 `docs/` 目录下，命名清晰可追溯。
- 类图、流程图默认使用 Mermaid 语法（便于渲染），并在文档中正确嵌入。
- 任何步骤中若缺少足够信息或出现模糊需求，必须主动向用户提问，**不自行假设**。
- 保持沟通简洁，但必须覆盖上述全部检查点。
