# 项目熟悉流程规范（C++ / Python 专用）

当用户要求你帮助熟悉一个 C++ 或 Python 项目时（例如"帮我梳理这个项目""分析下这个代码库"），你必须严格按照以下流程逐步分析，不得跳过或合并步骤。最终输出一份完整的、带有引用链接的"项目熟悉文档"。

## 适用范围与触发
- 适用语言：**C++ / Python** 项目。
- 触发场景：用户说"帮我梳理这个项目""分析下这个代码库""带我熟悉/看懂这个工程""onboard 一下"等。
- 产出：一份完整的、带引用链接的《项目熟悉文档》。

## 核心原则
- 必须严格按下方 8 步流程**依次执行，不得跳过或合并步骤**。
- 每一步完成后，如果需要用户确认某个理解（例如存在歧义的业务逻辑），**必须暂停并提问，不要自行假设**。

## 分析流程清单（必须依次执行）

复制以下清单并实时勾选进度：

```
- [ ] 1. 已有文档通读
- [ ] 2. 代码架构梳理
- [ ] 3. 重要技术细节挖掘
- [ ] 4. 业务逻辑梳理
- [ ] 5. 重要业务细节深挖
- [ ] 6. 其他细节补充（编码规范、工具等）
- [ ] 7. 缺陷评估
- [ ] 8. 输出最终文档
```

---

### 1. 已有文档通读
- 自动扫描项目根目录及子目录下的所有文档（README、`docs/` 中的 .md、wiki 链接、设计文档、CHANGELOG 等）。
- 提取并总结：项目目标、快速开始指南、构建/运行步骤、已知限制、贡献指南。
- 如果文档严重缺失，向用户指出，并说明后续分析将完全依赖代码。

### 2. 代码架构梳理
- 分析目录结构、模块/包划分、核心入口点（main 函数、主要类/模块）。
- **C++ 项目**特别关注：头文件与源文件组织、CMakeLists / 构建系统结构、命名空间使用、继承/多态体系。
- **Python 项目**特别关注：包导入关系、`__init__.py`、`setup.py`/`pyproject.toml`、Django/Flask 等框架中的 url 路由与 views 绑定。
- 输出架构概览，画出模块依赖关系图（使用 Mermaid 代码块，便于渲染）。

### 3. 重要技术细节挖掘
- 识别项目使用的关键库、框架、设计模式及底层技术。
  - 例如：C++ 中的 Boost.Asio、Qt 信号槽、RAII 与智能指针；Python 中的 asyncio、celery、pydantic 等。
- 对每个值得讲解的库/技术，说明**它被用在哪些地方、解决了什么问题、带来了哪些性能或维护性上的特点**。
- 对于非常重要的实现技巧（如模板元编程、协程调度、自定义装饰器等），要给出简要的原理解释。

### 4. 业务逻辑梳理
- 通过核心代码理解项目解决的业务问题，总结主要业务流程。
- 使用**流程图或时序图**（Mermaid）来表达关键业务流程，例如请求处理、数据处理管线、状态转换等。
- 描述清楚：谁触发、做什么、产生什么结果。

### 5. 重要业务细节深挖
- 关注影响正确性的关键细节：边界条件、异常处理、事务/回滚逻辑、幂等机制、权限校验、并发控制等。
- 解释这些细节为何重要，如果缺失可能导致什么问题。

### 6. 其他细节补充
- **编码规范**：命名惯例、缩进风格、注释习惯、文件组织偏好。
- **工具与配置**：构建工具（CMake、make、setuptools）、测试框架（gtest、pytest）、CI/CD 文件、Linter/Formatter 配置。
- **扩展机制**：插件加载、脚本扩展点、钩子函数等，方便后续开发。

### 7. 缺陷评估
- 基于以上分析，客观指出代码中可能存在的：
  - 技术债务（如过长函数、重复代码、过时 API 使用）
  - 性能瓶颈（同步阻塞、频繁内存分配、没必要的拷贝）
  - 安全隐患（SQL 注入风险、未检查的输入、硬编码密钥）
  - 可维护性问题（注释缺失、过度复杂的模板、循环依赖）
- 说明判断依据，而非给出笼统结论。

### 8. 输出最终文档
- 将 1~7 步的所有分析结果整合为一份完整的 Markdown 文档。
- 文档建议命名为 `项目熟悉文档-{项目名}.md`，并自然放置在项目 `docs/` 目录下（如该目录存在）。
- **文档中必须大量使用引用链接**，指向具体的文件、函数、行号，例如：
  - 已知代码框架：`主事件循环见 [event_loop.cpp](./src/core/event_loop.cpp#L45-L120)`
  - 业务逻辑：`下单流程见 [order_service.py](./services/order_service.py#L89)`
  - 重要库使用细节：`使用 libuv 实现异步 IO：[uv_wrapper.h](./src/net/uv_wrapper.h#L15)`
  - 引用时尽可能使用相对路径，并可精确到行号。

## 语言特性专项提醒
- **C++**：额外关注移动语义、完美转发、虚函数表、RAII 资源管理、编译时计算、链接依赖。
- **Python**：额外关注鸭子类型、描述符、协程、GIL 影响、import 副作用、包版本兼容性。

## 最终文档结构模板

第 8 步产出的《项目熟悉文档》建议采用以下章节骨架（文件名 `项目熟悉文档-{项目名}.md`）：

```
# 项目熟悉文档 - {项目名}

## 0. 速览（TL;DR）
- 一句话定位、语言/构建系统、技术栈、代码规模、如何跑起来

## 1. 已有文档概览
（对应流程第 1 步：现有文档总结 + 文档缺失情况）

## 2. 代码架构
（对应第 2 步：目录/模块划分、核心入口；附模块依赖关系图 Mermaid）

## 3. 关键技术与库
（对应第 3 步：关键库/框架/设计模式，用在哪、解决什么、有何取舍）

## 4. 核心业务流程
（对应第 4 步：主要业务流程，附流程图/时序图 Mermaid）

## 5. 关键业务细节
（对应第 5 步：边界条件、异常处理、事务、幂等、权限、并发等）

## 6. 工程细节
（对应第 6 步：编码规范、工具与配置、扩展机制）

## 7. 缺陷与风险评估
（对应第 7 步：技术债务 / 性能 / 安全 / 可维护性，逐条给出判断依据）

## 8. 待确认问题清单
（流程中暂停提问但尚未澄清的点，集中列出）
```

- 文中所有结论尽量挂上"相对路径 + 行号锚点"的引用链接（如 `[xxx.cpp](./src/xxx.cpp#L10-L42)`），便于读者跳转核对。
